# Open Controller Framework (OCF)

Open Controller Framework (OCF) is a modular, schema-first framework for defining, validating, transforming, and exporting controller systems in a target-agnostic way.

It is designed as the central logic layer between authoring tools such as FreeCAD and concrete targets such as DAWs, MIDI mappings, network protocols, or runtime environments.

## Status

Early stage.  
Architecture and documentation first.

This repository defines the conceptual and structural foundation of OCF before implementation details are expanded.

## Goal

OCF exists to make controller design portable, reusable, and formally describable.

Instead of coupling a controller directly to one runtime, one DAW, or one authoring tool, OCF introduces a canonical intermediate model for:

- controller definitions
- components and instances
- layouts
- constraints
- mappings
- interactions
- target adapters
- export pipelines

## Role in the Larger System

OCF is one part of a larger multi-language architecture:

- **C++ / Giada Fork**
  - audio / MIDI runtime
  - one possible runtime target
- **Rust / OCF**
  - core framework
  - canonical data model
  - layout and constraints
  - adapter and export logic
- **Python / FreeCAD Plugin**
  - mechanical authoring
  - primary CAD frontend
- **TypeScript / Tools / UI / Plugins**
  - external tools
  - editors
  - plugin workflows
  - ecosystem-facing integrations

## Important Clarification

OCF is **not** a Giada-specific framework.

Giada is only one possible adapter target.

OCF is intended to support multiple target classes, including:

- DAW integrations
- controller scripts
- MIDI mappings
- network-based protocols
- custom runtimes

OCF is also **not** defined by FreeCAD.

FreeCAD is the primary authoring frontend for mechanical design, but the core logic and canonical model must remain independent from any single frontend.

## Architecture Overview

OCF is structured around three major layers:

### 1. Core Logic

The core defines the portable, target-independent model:

- schemas
- types
- controller model
- layout rules
- constraints
- abstract interaction model
- validation and normalization

### 2. Adapter Layer

Adapters translate the generic OCF model into concrete targets.

Examples:

- Bitwig adapter
- Ableton adapter
- MIDI adapter
- Giada runtime adapter
- OSC or network adapter

### 3. Target / Runtime Layer

Targets consume the outputs generated by adapters.

Examples:

- DAW scripting environments
- MIDI mapping importers
- runtime engines
- networked control systems

## Adapter Model

An adapter is a target-specific translation layer.

It is responsible for:

- understanding target capabilities and limitations
- mapping abstract OCF structures to target concepts
- validating target compatibility
- generating export artifacts or handoff structures

The core must stay target-agnostic.  
Target-specific logic belongs in adapters.

## Workflow

The intended end-to-end workflow is:

**FreeCAD → Schema → OCF → Adapter → Target**

Example paths:

- FreeCAD → YAML → OCF → Bitwig script
- FreeCAD → YAML → OCF → MIDI mapping
- FreeCAD → YAML → OCF → Giada runtime integration

This keeps authoring, model logic, and target integration cleanly separated.

## Conceptual Target Classes

OCF is intended to support the following target classes over time:

- **DAW adapters**
  - Bitwig
  - Ableton
- **MIDI adapters**
  - generic MIDI mapping targets
- **Runtime adapters**
  - Giada
  - future custom runtimes
- **Network adapters**
  - OSC
  - other protocol-based targets

## Design Principles

- schema-first
- architecture-driven
- deterministic layouts
- modular adapters
- portable data model
- reusable component libraries
- strict separation of core and target logic

## Vision

OCF aims to become a reusable foundation for modular controller systems.

Long term, it should enable:

- reusable controller templates
- shared component libraries
- target-specific export pipelines
- mechanical + logical co-design
- a growing ecosystem of adapters, tools, and plugins

## Repository Focus

This repository is the logical heart of the system.

It should define:

- what a controller is
- how it is described
- how it is validated
- how it is transformed
- how it can be handed off to multiple targets

without collapsing into a single DAW, a single CAD frontend, or a single runtime.
